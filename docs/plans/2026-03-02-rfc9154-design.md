# RFC 9154 Compliance Design

**Date:** 2026-03-02
**RFC:** [RFC 9154 — Secure Authorization Information for Transfer](https://www.rfc-editor.org/rfc/rfc9154)
**Deadline:** Nominet production requires compliance from June 2026. Testbed already supports it.

## Background

RFC 9154 defines requirements for how EPP registrar clients generate and handle authorization information (auth codes) used in domain transfers. The current module uses `mt_rand` to generate 6–8 character auth codes, which provides at most ~47 bits of entropy — well below the 128-bit minimum the RFC mandates. Additionally, the RFC 9154 namespace must be declared in the EPP login command so the server applies the secure authinfo rules.

## Changes

### 1. `lib/epp_connection.php` — Register the RFC 9154 namespace

Add to `NominetEppConnection::__construct()` alongside existing Nominet extensions:

```php
$this->addExtension('secure-authinfo-transfer', 'urn:ietf:params:xml:ns:epp:secure-authinfo-transfer-1.0');
```

This causes the namespace to appear in the EPP login `<svcExtension>` block, signalling to the Nominet server that the client operates under RFC 9154 rules.

### 2. `nominet.php` — Compliant auth code generation

Replace the `generatePassword()` implementation with a CSPRNG-based alphanumeric generator producing 25-character codes (~148 bits of entropy, comfortably above the 128-bit minimum). Uses rejection sampling to eliminate modular bias.

```
pool  = a-z A-Z 0-9  (62 chars)
length = 25
entropy = log2(62^25) ≈ 148 bits

rejection threshold: accept bytes 0–247 (floor(256/62)*62 = 248),
reject 248–255 to ensure uniform distribution across the pool.
```

The method signature is preserved so existing call sites are unaffected. Length parameters are ignored — RFC 9154 compliance overrides any caller-specified length.

Both call sites benefit automatically:
- Domain registration auth code (`registerDomain`, line ~2018)
- Contact object auth code (`createContact`, line ~1996)
